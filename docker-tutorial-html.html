Docker Desktop Setup for Absolute Beginners

Welcome! This guide will walk you through setting up Docker Desktop on Windows, creating your first PostgreSQL database, and connecting to it with DBeaver. You'll also learn from my spectacular mistakes so you don't have to repeat them. No prior Docker knowledge required!

Before We Begin: Is Your Computer Ready?

Docker on Windows is like a sophisticated houseguest—it has some specific requirements. Let's make sure your system is ready to host it.

What You'll Need:

  • Windows 10/11 Pro, Enterprise, or Education (Sorry, Home edition users—Docker Desktop needs Hyper-V)
  • 64-bit processor with virtualization support
  • 4GB RAM minimum (8GB recommended for a smoother experience)
  • BIOS-level virtualization enabled (We'll check this together)

Your Task: Check Your Windows Version

1. Press Windows Key + R
2. Type winver and press Enter
3. Check that you have Windows 10 version 2004 or higher, or Windows 11

Check Virtualization Status:

1. Press Ctrl + Shift + Esc to open Task Manager
2. Click the "Performance" tab
3. Select "CPU" on the left
4. Look for "Virtualization: Enabled" on the right

⚠️ Important: If virtualization shows as "Disabled," you'll need to enable it in your BIOS. This requires restarting your computer and pressing a key (usually F2, F10, or Delete) during startup. Every computer is different, so Google "[your computer model] enable virtualization" for specific instructions.
If you have Windows Home edition, you can still use Docker with WSL2, but it's a bit more complex. Consider upgrading to Pro if you'll be doing serious development work. The upgrade usually costs around $99 and can be done directly through Windows Settings.

The Main Event: Installing Docker Desktop

Now that we've confirmed your system is ready, let's get Docker Desktop installed. This is easier than you might think!

Your Task: Download Docker Desktop

1. Go to Docker Desktop's official page
2. Click the big blue "Download for Windows" button
3. The installer is about 500MB, so grab a coffee while it downloads

Installation Steps:

1. Run the downloaded installer (Docker Desktop Installer.exe)
2. When prompted, ensure "Use WSL 2 instead of Hyper-V" is checked (recommended)
3. Click "Ok" and let the installation proceed
4. When it finishes, click "Close and restart" (yes, you need to restart)

⚠️ Common Issue: If you see "WSL 2 installation is incomplete," don't panic! Windows will prompt you to install WSL2. Just follow the prompts or visit the WSL2 installation page link it provides.

After Restart:

1. Docker Desktop should start automatically (look for the whale icon in your system tray)
2. You might see a tutorial—feel free to skip it for now
3. If prompted to sign in, you can skip this too (you don't need an account for local development)

If Docker Desktop doesn't start after restart, try right-clicking the Desktop shortcut and selecting "Run as administrator." Sometimes Windows needs a little extra permission nudge.

Creating Your First Database Container

Time to create something useful! We'll set up PostgreSQL in a container. Think of it as having a full database server that you can start, stop, or delete without affecting anything else on your computer.

Understanding docker-compose.yaml

Instead of typing long Docker commands, we'll use a configuration file. It's like a recipe that tells Docker exactly what we want.

Your Task: Create the Configuration

1. Create a new folder anywhere (e.g., C:\DockerProjects)
2. Inside that folder, create a new file called docker-compose.yaml
3. Open it with any text editor (Notepad works fine)
4. Copy this configuration:

version: '3.8'

services:
  postgres:
    image: postgres:15
    container_name: my_postgres
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: mydatabase
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
⚠️ Security Note: These are example passwords! In real projects, use strong passwords and never commit them to version control. We'll cover better security practices later.

Starting Your Database:

1. Open PowerShell or Command Prompt
2. Navigate to your project folder: cd C:\DockerProjects
3. Run: docker-compose up -d
4. Wait for the magic to happen!

You should see Docker downloading the PostgreSQL image (first time only) and then starting your container. The -d flag means "detached" - it runs in the background.

Verify It's Running:

Run: docker ps

You should see your postgres container listed with status "Up"!

If you get a "command not found" error, make sure Docker Desktop is running (check the system tray). If you see "no such file or directory," double-check you're in the right folder where you saved docker-compose.yaml.

Installing DBeaver: Because Command Lines Are So 1990s

Your PostgreSQL container is running, but how do you actually SEE your data? Enter DBeaver - a free, user-friendly database tool that works with pretty much everything.

Your Task: Download and Install DBeaver

1. Visit DBeaver's download page
2. Download the Windows installer (Community Edition is free and perfect for our needs)
3. Run the installer - accept all defaults, they're sensible
4. Launch DBeaver after installation

Connecting to Your PostgreSQL Container:

1. In DBeaver, click "New Database Connection" (the plug+ icon)
2. Select "PostgreSQL" and click Next
3. Fill in the connection details:

  • Host: localhost
  • Port: 5432
  • Database: mydatabase
  • Username: myuser
  • Password: mypassword

4. Click "Test Connection" - you should see "Connected"!
5. Click "OK" to save the connection

⚠️ Troubleshooting: If connection fails, make sure:
- Docker Desktop is running
- Your PostgreSQL container is up (`docker ps`)
- You're using "localhost" not "127.0.0.1" (some versions of Windows are picky)

Create Your First Table:

1. In DBeaver, expand your connection > Databases > mydatabase > Schemas > public
2. Right-click on "Tables" and select "Create New Table"
3. Name it "test_table" and add a few columns
4. Save and marvel at your containerized database!

DBeaver might prompt you to download PostgreSQL drivers on first connection. Just click "Download" and let it handle everything. It's surprisingly good at figuring out what it needs.

Docker Volumes: Handle with Care!

Time for a cautionary tale. Docker volumes are where your data lives outside containers. They're powerful, essential, and if you're not careful, very easy to accidentally destroy. Ask me how I know...

Understanding Volumes

Remember that volumes: section in our docker-compose.yaml? That creates a named volume called postgres_data. This is where PostgreSQL stores all your databases, tables, and data.

Why volumes matter:

  • Data persists even when containers stop or are removed
  • You can update containers without losing data
  • Multiple containers can share volumes (carefully!)

Useful Volume Commands:

View all volumes:

docker volume ls

Inspect a specific volume:

docker volume inspect postgres_data

See what's using a volume:

docker ps -a --filter volume=postgres_data
⚠️ THE DANGER ZONE - Read This Carefully! ⚠️
The command docker volume prune removes ALL unused volumes.
The command docker volume rm [volume_name] deletes a specific volume.
BOTH DELETE YOUR DATA PERMANENTLY. THERE IS NO UNDO.

Before running any volume deletion commands, always:
1. Run docker volume ls to see what you have
2. Back up any important data
3. Double-check you're deleting the right thing
4. Have a coffee and think about it one more time

My Personal Disaster Story:

When trying to add pgvector extension, an AI assistant suggested I "clean up volumes first" with docker volume prune -a. That little -a flag? It means "all volumes, even ones in use." Three days of test data: gone. Lessons learned:

  • AI assistants don't know your specific setup
  • Always understand what a command does before running it
  • Backups are not optional

Best Practices:

1. Name your volumes (like we did with postgres_data)
2. Back up regularly: docker exec my_postgres pg_dump -U myuser mydatabase > backup.sql
3. Think twice, delete once
4. Use .dockerignore to prevent accidental volume mounts of sensitive data

Pro tip: Create a simple backup script that runs pg_dump and saves with a timestamp. Run it before any major changes. Future you will thank present you when something inevitably goes wrong.

You Made It! Now Let's Keep You Safe

Congratulations! You've got Docker running, PostgreSQL containerized, and DBeaver connected. You've even learned from my mistakes. Let's wrap up with some best practices to keep your Docker journey smooth.

Security Best Practices:

  • Never use default passwords in production
  • Use environment variables for sensitive data: 
    POSTGRES_PASSWORD=${DB_PASSWORD}
  • Don't expose ports unnecessarily - only map what you need
  • Keep Docker Desktop updated - security patches matter

Performance Tips:

  • Limit container resources to prevent one container from hogging everything: 
    services:
  postgres:
    image: postgres:15
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
  • Use .dockerignore to exclude unnecessary files
  • Clean up unused images: docker image prune

Development Workflow:

  1. Version control your docker-compose.yaml (but not passwords!)
  2. Document your setup - future you will appreciate it
  3. Use consistent naming for containers and volumes
  4. Test locally before deploying

Common Commands Cheat Sheet:

# Start your stack
docker-compose up -d

# Stop everything
docker-compose down

# See what's running
docker ps

# Check logs
docker logs my_postgres

# Execute commands in container
docker exec -it my_postgres psql -U myuser

# Emergency stop all containers
docker stop $(docker ps -q)

Where to Go From Here:

Now that you have the basics, consider exploring:

  • Adding more services (Redis, RabbitMQ, etc.)

  • Container orchestration with Docker Swarm or Kubernetes

Remember: Everyone was a beginner once. The Docker community is generally helpful - don't be afraid to ask questions. Just maybe double-check any commands involving the word "prune" before running them! 😄
